# EAP 平台實作流程

> 從登入到跑出第一個 Demo QA 的完整步驟。
> 平台：Gemini Data EAP（cloud.geminidata.com）

---

## 步驟 0：登入並建立專案

1. Chrome 開啟 `https://cloud.geminidata.com`
2. 登入後 → **改密碼**（若還是預設 `Gemini12345!`）
3. 左上角專案下拉 → 「**+ 新建專案**」
4. 命名建議：`職災法規助手_PoC`（之後正式版另開 `_v1`）

---

### 操作要點

- 若有檔案失敗 → 看錯誤訊息（多半是檔名特殊字元、檔案太大）

## 步驟 2：建立 Data Source（CSV）

Data Importer → **+ Import Data**（不是 Import Documents）

### 上傳順序：先節點、後關係

#### 2A. 7 個節點 CSV

每個節點 CSV 都是一個 Data Flow，步驟相同：

1. **+ Add Source Data** → 上傳一個節點 CSV（如 `01_事故.csv`）
2. 系統解析後在 Models 區建立對應 Model
3. 進 Model 編輯：
   - **Category** 命名（例：事故類型）
   - **Unique Key** 選擇對應 ID 欄位（例：事故ID）
   - 其他欄位作為 properties
4. **Save Model** → 回到 Flow
5. **按 Start** ─ Source + Model 寫進畫布
6. **檢查 Status** 是否為「Success」

完整對應表：

| CSV 檔 | Category | Unique Key |
| --- | --- | --- |
| 01_事故.csv | 事故類型 | 事故ID |
| 02_法規條文.csv | 法規條文 | 條文ID |
| 03_企業義務.csv | 企業義務 | 義務ID |
| 04_勞工權益.csv | 勞工權益 | 權益ID |
| 05_文件.csv | 文件 | 文件ID |
| 06_罰則.csv | 罰則 | 罰則ID |
| 07_受理單位.csv | 受理單位 | 單位ID |

#### 2B. 5 個關係 CSV

關係 CSV 也是建 Data Flow，但 Model 設定不同：

1. 上傳關係 CSV（例：`R1_事故適用法規.csv`）
2. 進 Model 編輯：
   - 兩個 ID 欄位都要**指定為「連到既有節點」**
   - 第一欄（事故ID）→ 連到 Category=事故類型 的節點
   - 第二欄（條文ID）→ 連到 Category=法規條文 的節點
   - **關係類型命名**（例：`適用於`）
3. Save → Start

完整對應表：

| CSV 檔 | 起點 Category | 終點 Category | 關係類型建議命名 |
| --- | --- | --- | --- |
| R1_事故適用法規.csv | 事故類型 | 法規條文 | 適用於 |
| R2_法規衍生義務.csv | 法規條文 | 企業義務 | 衍生義務 |
| R3_法規衍生權益.csv | 法規條文 | 勞工權益 | 衍生權益 |
| R4_義務需文件.csv | 企業義務 | 文件 | 需文件 |
| R5_權益需文件.csv | 勞工權益 | 文件 | 需文件 |

> ⚠️ 工作坊講義第 1140 行強調：**編輯 Data Flow 後務必「清空畫布」再全部 Data Flow 重跑**！

## 步驟 3：視覺化檢查

進 Explore（畫布探索） → 拉出 Category=`事故類型` 的所有節點 → 觀察是否：

- A001 通勤事故 → 連到 4 條法規（L001, L003, L004, L005）
- A002 工地墜落 → 連到 6 條法規（含 L007 重大職災通報）

可對照 [`資料/情境預覽.txt`](資料/情境預覽.txt) 的預期遍歷結果。

## 步驟 4：設定 Robot（機器人系統提示）

進 Q&A → 機器人設定 → 貼入下方 prompt：

```text
你是「職災法規智慧助手」，依使用者身分（人資 / 勞工）回答事故對應的法定義務或權益。

回覆規則：
1. 使用繁體中文，台灣用語。
2. 回答前請先確認身分（如使用者未表明，先問：「請問您是企業人資還是勞工/工會？」）。
3. 若身分為「人資」：依下面結構輸出：
   ## 法定義務
   - 義務名稱（依據條文）｜申報期限｜通報單位
   ## 應備文件
   - 文件名稱（用途）
   ## 違規罰則
   - 罰則名稱｜罰鍰範圍｜處分單位
   ## 後續調解程序
4. 若身分為「勞工」：依下面結構輸出：
   ## 可請領給付
   - 給付名稱（時效）｜申請條件｜受理單位
   ## 應備文件
   - 文件名稱（取得來源）
   ## 申訴調解
5. 引用條文時請標明法規名稱與條號（如：勞基法§59、災保法§42）。
6. 若資訊不足以判定為職災，回應：「需進一步釐清以下資訊：...」並列出待確認項目（事故時間/地點、是否上下班途中、是否屬重大職災等）。
7. 每次回答最後列出 3 個延伸問題範例。

Generate Cypher 時使用 Neo4j v5 語法，不要使用 OVER()。
```

## 故障排除

| 症狀 | 原因 | 解法 |
| --- | --- | --- |
| 畫布空空但 Status 顯示 Success | 瀏覽器快取 | 登出再登入 |
| 兩個事故節點 ID 一樣但不合併 | Category 沒對齊 | 確認 7 個節點檔的 Category 命名沒寫錯字 |
| 關係邊只連一邊 | 關係 Model 沒指定終點 Category | 重新編輯 Model，按 Save → 清空畫布 → 全部重跑 |
| QA 回答「找不到資料」 | Robot Setting 沒存 / 文件還在處理中 | 檢查 Documents 處理狀態、Robot Setting 是否點 Save |
| 數字欄位變成字串 | CSV 欄位有單位文字 | 重新編輯 CSV，移除 "30 萬" 改 `300000` |
