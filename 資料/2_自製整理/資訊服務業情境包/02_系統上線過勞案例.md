# 系統上線長工時 / 過勞風險案例

## 情境摘要

某資訊服務公司承接大型客戶核心系統改版。專案進入上線前兩個月後，工程師長期處於高壓狀態，每週多次夜間部署、假日驗證與凌晨 incident 排除。工程師後續因心血管疾病住院，或更嚴重地發生猝死事件。

這是資訊服務業最適合作為主 demo 的案例，因為它同時牽涉專案時程、加班管理、on-call 制度、健康風險、職業病認定、雇主補償、勞保給付與公司治理責任。

## RAG 系統要解決的問題

單純查法條只能知道「有職災補償」或「有異常工作負荷指引」，但企業真正需要的是：

- 是否已達異常工作負荷風險訊號？
- 哪些資料可以支撐工時與工作負荷判斷？
- HR、PM、主管、EHS、法務要分別做什麼？
- 如果員工住院或死亡，要如何切換到職災補償與給付流程？
- 公司是否有制度性風險，例如長期上線人力不足、on-call 沒有正式紀錄、加班申請與實際工作時間不一致？

## 需要蒐集的證據

| 證據 | 用途 |
| --- | --- |
| 出勤紀錄 | 判斷實際工作日數、每日工時、連續出勤 |
| 加班申請與核准紀錄 | 比對正式加班與實際工作負荷 |
| on-call 值班表 | 判斷夜間與假日待命是否常態化 |
| incident ticket | 證明夜間、假日或緊急維運工作內容 |
| 系統上線紀錄 | 連結專案時程與工作壓力來源 |
| 專案會議與訊息紀錄 | 證明客戶壓力、時程變更、臨時需求 |
| 診斷證明 | 連結健康結果與後續給付或補償流程 |
| 健康檢查 / 醫師評估紀錄 | 評估高風險員工與雇主預防措施 |

## 對應法規與指引

- 職業安全衛生設施規則第 324-2 條：異常工作負荷促發疾病預防措施
- 勞動基準法第 30 條：工時與出勤紀錄
- 勞動基準法第 32 條：延長工時與加班限制
- 勞動基準法第 36 條：例假、休息日與連續出勤限制
- 勞工職業災害保險及保護法：傷病、失能、死亡給付
- 職業促發腦心血管疾病認定參考指引
- 異常工作負荷促發疾病預防指引

## Demo 問句

```text
工程師連續兩個月配合客戶系統上線，常常半夜處理 incident，現在因心血管疾病住院。HR 應該先看哪些資料？可能涉及哪些法規與處置？
```

## 預期回答重點

1. 先提醒需要判斷是否屬於異常工作負荷與可能職業促發疾病。
2. 要求蒐集出勤紀錄、加班申請、on-call 表、incident ticket、系統上線紀錄與診斷證明。
3. 法規面連到異常工作負荷預防、工時、加班、例休與職災給付。
4. 流程面要求 HR 先啟動職災初判，PM 補專案與 incident 資料，EHS 評估工作負荷，法務評估補償與爭議風險。
5. 商業風險面指出若公司長期沒有值班制度、工時紀錄或健康風險管理，會形成內控制度缺口。

## 商業價值

這個案例能把產品定位從「查職災法條」提升成「專案風險與勞動合規決策輔助」。對資訊服務公司來說，這比工地墜落更貼近真實營運。
