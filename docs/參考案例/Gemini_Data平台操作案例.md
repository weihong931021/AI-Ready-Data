# Gemini Data 平台操作案例觀察

> 來源：使用者提供的別組簡報截圖。此文件用來記錄可借鏡的呈現方式，不代表該作品完整內容。

## 作品主題

截圖中的作品以雙品牌銷售資料作為案例，使用 Gemini Data 平台完成資料匯入、知識圖建模、節點關係設定，最後讓 AI 產生銷售分析報表與營運洞察。

核心資料欄位包含：

- Order ID
- Month / Year
- Brand
- Location / StoreType
- Customer ID
- Nationality / Type
- Net Sales (USD)

## 平台操作流程

### 1. 資料清理與匯入

簡報把資料前處理講得很具體：

- 篩選 Calvin Klein 和 Tommy Hilfiger 兩個品牌資料。
- 修正月份錯字，例如 `Janurary` 改為 `January`。
- 設定 `Order ID` 與 `Customer ID`。
- 將 Tommy 的 ID 從 10001 開始，避免和另一品牌資料混淆。
- 依年份與品牌拆分資料，例如 2023 CK、2024 CK、2023 Tommy、2024 Tommy。
- 統一 `Net Sales (USD)` 為小數點後兩位。

可借鏡點：資料清理不是只說「已清理」，而是列出具體修正規則，讓評審知道資料有被處理過。

### 2. 知識圖繪製

該作品把銷售資料拆成節點：

- 訂單 ID
- 顧客 ID
- 品牌
- 時間年月
- 門市 / 地點
- 店鋪型態
- 國籍
- 消費者類型
- 銷售額

關係設計包含：

- 顧客購買品牌
- 訂單發生於時間
- 訂單產生銷售額
- 訂單發生於門市
- 顧客來自國籍
- 門市屬於店鋪型態

可借鏡點：即使資料本身是表格，也要先用圖示講清楚「哪個欄位變成節點、哪個欄位變成關係」。

### 3. 設定節點與關係

簡報直接放 Gemini Data 平台畫面，展示：

- Step 2-2: Edit Node(s)
- 每個 Node 的 Category 設定
- Node 內欄位如何對應
- 關係名稱，例如 `ORDERED_IN_YEAR`、`FOR_BRAND`、`PLACED_BY`、`PURCHASED_AT`、`HAS_NATIONALITY`

可借鏡點：評審容易理解平台操作是否真的做過；我們的提案也應放 Data Model / Data Flow / Explore 畫面，而不是只畫概念圖。

## Demo 呈現方式

該作品的 Demo 問題集中在營運分析：

- 2023/2024 年兩品牌銷售高峰與最低銷售額月份。
- 分析高峰期與低谷期原因，並提出促銷或行銷建議。
- 找出 CK 與 Tommy 的主要消費者國籍，計算平均消費金額、訂單數與訂單總金額。
- 找出具備擴大營運潛力的前五個門市地點，以及應縮減資源的地點。
- 比較 FSS / SIS 店鋪類型的消費金額與訂單數。

可借鏡點：Demo 問題不是單純查資料，而是「查詢 + 分析 + 建議」。我們的職災專案也應把問題設計成：事故判斷、法規義務、裁罰風險、下一步建議。

## 對本專案的啟發

### 可以直接學的部分

- 簡報中安排連續三頁平台操作：資料清理與匯入、知識圖繪製、設定節點與關係。
- 每個節點/關係都用具體欄位與關係名稱說明。
- Demo 問題要用真實業務語氣，而不是資料庫查詢語氣。
- 最後成果要有圖表、AI 回答、知識圖畫面三種證據。

### 本專案應強化的對應頁面

- 資料清理：說明 26,513 筆裁罰資料如何整合來源、日期、金額、法條、產業。
- 知識圖：展示事故、法規、義務、權益、文件、刑責、民賠、治理風險的節點設計。
- 節點關係：展示 R1-R13，其中 R9/R12/R13 拆開是為了符合 Gemini Data 單一起點 Category 的建模限制。
- Demo：以「工地墜落重大職災」為主，輸出企業通報/補償/刑責/民賠/治理風險，再用勞工視角補權益。
